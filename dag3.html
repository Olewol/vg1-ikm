🧠 Teori – Brukeradministrasjon

📚 Les og forstå – ikke noe som skal leveres inn

🔒
Principle of Least Privilege – En bruker skal bare ha akkurat den tilgangen de trenger for å gjøre jobben sin. Dette er et grunnleggende sikkerhetsprinsipp i all IT-administrasjon.
BrukertypeBeskrivelse
AdministratorFull kontroll. Kan installere programmer og endre systeminnstillinger.
StandardbrukerKan bruke programmer og endre egne innstillinger, men ikke installere software.
GjestSvært begrenset tilgang. Vanligvis deaktivert av sikkerhetsgrunner.

Innebygde brukergrupper i Windows

GruppeBeskrivelse
AdministratorsFull kontroll over systemet. Kan installere alt og endre alle innstillinger.
UsersStandard brukere. Kan logge inn og bruke installerte programmer.
GuestsSvært begrenset tilgang. Kontoen deaktiveres normalt av sikkerhetsgrunner.
Remote Desktop UsersKan logge inn via Fjernskrivebord uten å være lokal administrator.
PowerShell / CMD
net user                          # Vis alle brukere
net user <navn> <passord> /add    # Opprett bruker
net user <navn> /delete           # Slett bruker
net localgroup Administrators <bruker> /add  # Admin-gruppen
Bash
sudo adduser <brukernavn>          # Opprett bruker
sudo deluser --remove-home <navn>  # Slett med hjemmappe
sudo passwd <brukernavn>          # Endre passord
sudo usermod -aG sudo <bruker>   # Gi sudo-rettigheter
cat /etc/passwd                    # Vis alle brukere
su - <brukernavn>                  # Bytt til bruker
💡
sudo lar deg utføre enkeltkommandoer med administratorrettigheter uten å være logget inn som root. Tryggere enn å jobbe som root.

☁️ Sentral administrasjon – Microsoft Entra ID

📚 Les og forstå – ikke noe som skal leveres inn

🔵
Microsoft Entra ID (tidligere kalt Azure AD) er Microsofts skybaserte brukeradministrasjonstjeneste. De fleste norske skoler bruker det gjennom Microsoft 365 – det er her du logger inn med @skole-e-postadressen din.
BegrepForklaring
TenantOrganisasjonens Entra ID-instans – skolens «sky-domene»
BrukerKonto som kan logge inn og bruke ressurser (f.eks. Microsoft 365)
GruppeSamling av brukere for felles tilgangsstyring
RolleDefinerer hva en bruker/gruppe kan administrere (f.eks. Global Administrator)
MFAMulti-faktor autentisering – ekstra sikkerhetssjekk ved innlogging
Conditional AccessRegler for hvem som kan logge inn fra hvilke enheter og lokasjoner
LisensMicrosoft 365-lisens kobles til brukerkontoen og gir tilgang til Teams, Word osv.

Oppgave 3.1 – Lokale brukere i Windows (GUI)
Åpne Datamaskinbehandling → Lokale brukere og grupper → Brukere. Opprett: elev_anna (standard), elev_bjorn (standard), it_admin (legg til i Administrators). Passord: Passord123!. Logg inn som elev_anna og observer begrenset tilgang.
💡
Åpne Datamaskinbehandling raskt: Trykk Win + R, skriv compmgmt.msc og trykk Enter.

Logg inn som en annen bruker: Klikk Start → profilikon → «Bytt bruker» (Switch account). Du trenger ikke logge ut – du kan bytte mellom brukere.

elev_anna klarer ikke installere? Det er meningen! Standardbrukere får UAC-dialog der de må oppgi administratorpassord. Dette er Principle of Least Privilege i praksis.
🟢
Oppgave 3.2 – PowerShell-brukeradministrasjon
⚠️
Kjør PowerShell som administrator!
Kommandoene under krever administratorrettigheter. Høyreklikk på PowerShell i Start-menyen og velg «Kjør som administrator» (Run as administrator). Ser du en UAC-dialog? Det er Windows som bekrefter at du vet hva du gjør – klikk Ja.
PowerShell (Kjør som administrator)
net user                                    # 1. Vis alle brukere
net user testbruker Passord123! /add        # 2. Opprett bruker
net user testbruker                         # 3. Se detaljer
net localgroup Users testbruker /add        # 4. Legg i Users-gruppe
net user testbruker /delete                 # 5. Slett brukeren
net user                                    # 6. Bekreft sletting
🟢
Oppgave 3.3 – Brukere i Ubuntu
Bash
sudo adduser praksis_elev                  # 1. Opprett bruker
cat /etc/passwd | grep praksis_elev        # 2. Verifiser
sudo usermod -aG sudo praksis_elev        # 3. Gi sudo-rettigheter
groups praksis_elev                        # 4. Sjekk gruppemedlemskap
su - praksis_elev                          # 5. Bytt til bruker
whoami                                     # 6. Hvem er jeg?
sudo ls /root                             # 7. Test sudo
exit                                       # 8. Gå tilbake
sudo deluser --remove-home praksis_elev   # 9. Slett
🟢
Oppgave 3.4 – Delt mappe med tilgangsstyring
Bruk brukerne du opprettet i oppgave 3.1 og sett opp en mappe med NTFS-rettigheter.

  1. Opprett mappen C:\FellesMappe

  2. Høyreklikk mappen → Egenskaper → fanen Sikkerhet

  3. Klikk Rediger → Legg til → skriv inn it_admin og bekreft
    Gi it_admin: Full kontroll

  4. Legg til elev_anna og elev_bjorn på samme måte
    Gi dem: Endre (kan lese og skrive, men ikke endre rettigheter)

  5. Fjern Alle / Everyone fra listen hvis den finnes

  6. Klikk Bruk og OK

  7. Logg inn som elev_anna – kan du opprette en fil i C:\FellesMappe?

  8. Logg inn som it_admin – kan du slette filer som elev_anna opprettet?

💡
NTFS-rettigheter gjelder på filsystemnivå, uavhengig av deling. Det er god praksis å alltid sette NTFS-rettigheter i tillegg til delingsrettigheter for å ha full kontroll over hvem som har tilgang.

Bonusoppgave 3.G – Passordpolicy med Local Security Policy
Windows lar deg tvinge frem sterke passord via innebygde sikkerhetsregler – akkurat slik det gjøres i bedrifter og skoler.

  1. Trykk Win + R, skriv secpol.msc og trykk Enter
  2. Gå til Account Policies → Password Policy
  3. Sett Minimum password length til 10
  4. Aktiver Password must meet complexity requirements
  5. Klikk OK og lukk vinduet
  6. Prøv å sette passordet abc på elev_anna via PowerShell:
    net user elev_anna abc – hva skjer?
  7. Sett i stedet et passord som oppfyller kravene, f.eks. Skole2025!

Svar på:
  • Hva er «password complexity requirements» i Windows?
  • Hvem bør bestemme passordpolicy i en organisasjon?
🔵
Bonusoppgave 3.H – Event Viewer: finn hvem som logget inn
Windows logger alle innloggingsforsøk automatisk. I en virkelig jobb brukes dette til å oppdage uautorisert tilgang.

  1. Trykk Win + R, skriv eventvwr.msc og trykk Enter
  2. Gå til Windows Logs → Security
  3. Se etter hendelser med Event ID 4624 (vellykket innlogging) og 4625 (feilet innlogging)
  4. Klikk på en 4624-hendelse og les detaljene: hvilken bruker logget inn, og når?
  5. Logg inn og ut som elev_anna, gå tilbake til Event Viewer og finn hendelsen

Svar på:
  • Hva viser Event ID 4624 og 4625?
  • Hvorfor er revisjonslogg (audit log) viktig i en bedrift?
  • Hva er skjermbildet av hendelsen du fant?

🧠 Teori – Nettverksmodi

📚 Les og forstå – ikke noe som skal leveres inn

🔹
NAT
VM deler vertens IP mot internett. Usynlig utenfra. Bruk: trenger internett, ikke synlighet.
🔹
Bridged
VM får egen IP på samme nettverk som verten. Synlig for andre. Bruk: separat enhet på nettverket.
🔹
Host-only
VM kan kun kommunisere med vert og andre VM-er. Ingen internett. Bruk: isolert lab-miljø.

IP-adresser og subnetting

En IP-adresse er som en postadresse for en datamaskin – fire tall mellom 0 og 255, atskilt med punktum. Private adresseområder brukes internt i nettverk:

KlasseEksempelTypisk bruk
Privat A10.0.0.0/8Store bedriftsnettverk
Privat B172.16.0.0/12Mellomstore nettverk
Privat C192.168.0.0/24Hjemmebruk og lab
BegrepForklaringEksempel
IP-adresseUnik nettverksadresse for en enhet192.168.1.10
SubnettmaskeAngir nettverks- vs. enhetsdelen255.255.255.0 (/24)
GatewayIP til ruteren – utgangen fra nettverket192.168.1.1
DNSOversetter domenenavn til IP-adresser8.8.8.8 (Google)
DHCPAutomatisk IP-tildelingRuteren gir deg IP automatisk

Ping – nettverkstesting

SHELL
# Test internettforbindelse
ping 8.8.8.8

# Test lokal gateway
ping 192.168.1.1

# Test kommunikasjon mellom VM-er
ping 192.168.100.20

# Ubuntu: begrens til 4 pakker
ping -c 4 8.8.8.8
🎧
I VMware har du tre nettverksmodi. NAT gir internettilgang via verten. Bridged gir VM-en egen IP og gjør den synlig på nettverket. Host-only isolerer VM-en – den kan kun snakke med verten. Alle enheter har en IP-adresse. ping tester om to enheter kan kommunisere.

💻 Oppgaver – Dag 2

📎

🟢
Oppgave 2.1 – NAT og internettilgang (Økt 1 – begge klasser)
Mål: Verifisere at VM-en har internettilgang via NAT.

  1. Åpne Ledetekst (CMD) eller PowerShell i VM-en

  2. Kjør: ipconfig – noter IP-adresse, subnettmaske og gateway

  3. Kjør: ping 8.8.8.8 – sjekk at du får svar

  4. Kjør: ping google.com – sjekk at DNS fungerer

  5. Åpne nettleseren og besøk en nettside

  1. Åpne Terminal (Ctrl+Alt+T)

  2. Kjør: ip addr – finn IP-adressen

  3. Kjør: ping -c 4 8.8.8.8

  4. Kjør: ping -c 4 google.com

📸
Krav: Skjermbilde av ipconfig/ip addr output · Skjermbilde av vellykket ping · Tegn en enkel skisse: VM → VMware NAT → Vertsmaskin → Internett
🟢
Oppgave 2.2 – Host-only nettverk (Økt 2 / Klasse B: Dag 3 økt 1)
Mål: Opprette isolert nettverk mellom to VM-er. Krever to kjørende VM-er.

  1. Sett begge VM-ene til Host-only (VM Settings → Network Adapter → Host-only)

  2. Start begge VM-ene

  3. Sjekk IP-adressene: Windows: ipconfig | Ubuntu: ip addr

  4. Ping fra VM 1 til VM 2, og fra VM 2 til VM 1

📸
Krav: Skjermbilder av begge IP-adresser · Ping vellykket i begge retninger · Forklar: Hvorfor har VM-ene ikke internettilgang nå?
🟢
Oppgave 2.3 – Statisk IP-adresse (Økt 2 / Klasse B: Dag 3 økt 1)
Mål: Sette en statisk (fast) IP-adresse manuelt.

  1. Høyreklikk nettverksikonet → Åpne nettverksinnstillinger

  2. Ethernet-adapteret → Egenskaper → Internet Protocol Version 4 (TCP/IPv4)

  3. Velg «Bruk følgende IP-adresse»:
    IP: 192.168.100.10 · Maske: 255.255.255.0 · Gateway: 192.168.100.1

  4. Klikk OK og verifiser med ipconfig

  1. Klikk nettverksikonet → Wired Settings → tannhjulikonet

  2. IPv4-fanen → bytt fra Automatic (DHCP) til Manual

  3. Fyll inn:
    Address: 192.168.100.20 · Netmask: 255.255.255.0 · Gateway: 192.168.100.1

  4. Klikk Apply og verifiser med ip addr

💡 Tips og feilsøking

🔥
Ping fungerer ikke – vanlige årsaker:
  • Sjekk at nettverksadapteren i VM-en er satt til NAT (ikke Host-only eller Bridged)
  • Windows-brannmuren kan blokkere ping. Test med ping 8.8.8.8 – hvis det fungerer, men ikke ping google.com, er det DNS-problemet
  • Prøv å slå av og på nettverksadapteren i VM-en: Nettverksinnstillinger → deaktiver → aktiver adapteren
🌐
Statisk IP – mistet internett?
Når du setter statisk IP mister du internett hvis gateway eller DNS mangler. Sjekk:
  • Windows: Gateway og DNS må fylles inn – ikke la dem stå tomme. Bruk 8.8.8.8 som DNS
  • Ubuntu: Etter å ha trykket Apply – toggle nettverkstilkoblingen av og på for at innstillingene skal tre i kraft
  • Vil du tilbake til automatisk IP: velg DHCP igjen og toggle tilkoblingen
🔗
Host-only – VM-ene finner ikke hverandre?
  • Sjekk at begge VM-ene er satt til Host-only – ikke bare én
  • Sjekk IP-adressene med ipconfig / ip addr – begge må være i samme subnett (f.eks. 192.168.56.x)
  • Windows: Brannmuren kan blokkere ping mellom VM-er. Prøv å slå av Windows Defender Firewall midlertidig for å teste

Fordypningsoppgave 2.F – Subnetting
Du har nettverket 192.168.50.0/24. Del det i 4 like store subnett. For hvert subnett: hva er subnettmasken, nettverksadresse, første og siste brukbare adresse, og broadcast-adresse?
Se oppslagsverk → Subnettmaske-tabell for hjelp.
🔵
Bonusoppgave 2.G – Rutetrace: se pakkenes vei
Bruk tracert (Windows) eller traceroute (Ubuntu) for å se hvilke rutere pakkene passerer på vei til internett.

CMD / PowerShell
tracert google.com
Bash
traceroute google.com
Svar på:
  • Hvor mange hopp (hops) er det til google.com?
  • Hva er IP-adressen til det første hoppet? (Det er din gateway/ruter)
  • Hva skjer med responstiden (ms) desto lenger unna du kommer?
🔵
Bonusoppgave 2.H – Bridged adapter: VM på det ekte nettverket
Bytt nettverksmodus til Bridged og se hva som endrer seg.

  1. Gå til VM Settings → Network Adapter → sett til Bridged
  2. Restart nettverksadapteren i VM-en (eller restart VM-en)
  3. Kjør ipconfig (Windows) / ip addr (Ubuntu) – noter den nye IP-adressen
  4. Sammenlign: Er VM-ens IP nå i samme subnett som vertsmaskinens IP?
  5. Prøv å pinge vertsmaskinens IP fra VM-en
  6. Bytt tilbake til NAT når du er ferdig
